# /goal-mode Pre-flight Gates

`/ship-execute` runs as /goal-shaped by default — it works from sprint-start to sprint-done without per-wave user hand-off, halting only on the documented escalation contracts. The gates below refuse to enter that mode when a known-ambiguous condition exists, so a /goal run never compounds a structural problem.

Run each gate in order. The first failure emits `sprint_goal_preflight_failed` via `shipyard-data events emit sprint_goal_preflight_failed gate=<name> sprint=<id>` and halts with an actionable message. Skip on `--task` and `--hotfix` modes — those are scoped and don't run /goal-shaped.

## The gates

| Gate | Check | Why | Actionable fix |
|---|---|---|---|
| `sprint_plan_accepted` | SPRINT.md frontmatter `status: planned` or `status: in-progress` (not `proposed`) | A plan not accepted may be wrong; running it as /goal compounds the wrong-ness | `/ship-sprint --accept` or edit SPRINT.md frontmatter |
| `every_feature_has_ac_signal` | Each feature in SPRINT.md has at least one `acceptance_probe:`-bearing task OR a `demo_probe:` on the feature | Without acceptance signals, the sprint-complete predicate is undefined — invariant 4 (orphan AC) becomes vacuously true | `/ship-discuss <feature-id>` to add probes |
| `no_open_spec_ambiguities` | grep linked feature/AC files for `ambiguity:` markers — none must be present | These are the `design_ambiguity` escalation in disguise; surface them now, not after /goal halts | Resolve via `/ship-discuss <feature-id>` or remove the marker |
| `sprint_verify_defined` | `config.md` has `test_commands.full` (or an equivalent sprint-level verify command) set | Without it, sprint-complete invariant 2 (sprint-boundary probe) is undefined | Edit `<SHIPYARD_DATA>/config.md` to add `test_commands.full` |
| `working_branch_correct` | Current branch equals SPRINT.md `branch:` (existing Pre-spawn Branch Check, lifted earlier) | Catches the worktree-branch-leak case before any subagent spawns | `git checkout <sprint-working-branch>` |
| `no_active_discussion_lock` | `.active-session.json` not held by a live different session (existing `acquiring-skill-lock` check) | Discussion-in-flight conflicts with execution | Wait for the other session to release, or force-clear if known-dead |
| `event_log_initialized` | `<SHIPYARD_DATA>/.shipyard-events.jsonl` exists | Required for /goal observability and resume-from-event-log | Run `/ship-init --repair` or emit any event to bootstrap |

## Halt protocol

All gates green → proceed. Any gate red → emit the failure event, show the user the failing gate name plus the actionable fix from the table, and halt. Do NOT proceed in degraded /goal mode — partial /goal is worse than user-driven execution because the partial bits hide real failure modes.

## Why this exists

/goal-mode is /ship-execute's most powerful surface and its most opaque. Without pre-flight gates, the orchestrator could enter /goal on a plan that's missing acceptance signals, hit a wall mid-flight, and surface a confusing failure to the user — *"why did /goal halt? I don't see what's broken."*

The gates frame the halt up front with concrete remediation, before any tokens get spent on actual work. Users see "Pre-flight failed: sprint_verify_defined — add `test_commands.full` to config.md" instead of "Sprint-complete predicate failed at invariant 2 after 4 hours of execution."

## Per-gate detail

### `sprint_plan_accepted`

A sprint in `status: proposed` was generated by `/ship-sprint` but the user hasn't reviewed and accepted it. /goal-mode assumes the plan is the intended thing to ship — running an un-accepted plan to completion is a near-certain misuse.

The check is a one-line YAML frontmatter read; no CLI primitive needed.

### `every_feature_has_ac_signal`

For each `feature_id` in SPRINT.md, read the feature file and its linked task files. The feature passes if EITHER:

- One or more linked tasks has `acceptance_probe:` set in its frontmatter, OR
- The feature itself has `demo_probe:` set in its frontmatter.

The two probe layers are: per-task probes prove wiring works within the task; per-feature demo probes prove the cross-task user flow composes. /goal-mode needs at least one for the sprint-complete predicate to have a falsifiable signal.

### `no_open_spec_ambiguities`

Recursive grep for `ambiguity:` markers in:

- Every feature file in this sprint
- Every task file linked to those features
- Every AC entry referenced by those features

Spec ambiguities are intentional placeholders — "this AC needs clarification before we ship." A subagent encountering one would return `ESCALATION_CODE: design_ambiguity`, halting the wave. Catching them at pre-flight is cheaper than halting mid-sprint.

### `sprint_verify_defined`

The sprint-boundary verify-probe is broader than wave-probes — it's the full-suite or integration check. Without it, sprint-complete invariant 2 has nothing to test, so invariant 2 is undefined and the predicate is undermined.

Read `<SHIPYARD_DATA>/config.md` and check for `test_commands.full` (or any equivalent key the project uses). The check is structural — content of the command doesn't matter, just that one exists.

### `working_branch_correct`

`git branch --show-current` against SPRINT.md's `branch:` field. Already done by the existing Pre-spawn Branch Check in /ship-execute Step 2; lifted to pre-flight because the cost of getting this wrong (worktrees branch from the wrong commit, hours of merge work to recover) is higher than the cost of one extra check.

### `no_active_discussion_lock`

Reads `<SHIPYARD_DATA>/.active-session.json`. If held by a different session (`session_id` differs) AND the timestamp is recent (within the staleness threshold), refuse. The `acquiring-skill-lock` capability skill owns the canonical lock-checking logic — this gate just invokes it.

### `event_log_initialized`

`<SHIPYARD_DATA>/.shipyard-events.jsonl` must exist (can be empty). Required because:

- /goal-mode emits heartbeat events; `events emit` will create the file if missing but the gates run before the first emission.
- The resume-from-event-log protocol reads the log to find the last clean checkpoint. An empty file is fine; a missing file means resume is impossible.

If missing, run `shipyard-data events emit goal_preflight_check` to bootstrap it (or any other no-op event).
