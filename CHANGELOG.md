# Changelog

All notable changes to Shipyard are documented here. Format follows [Keep a Changelog](https://keepachangelog.com/).

## [1.0.4] - 2026-04-06

**Fix: user-owned worktrees now get isolated data dirs.**

### Fixed
- **Independent Claude sessions in separate user-owned worktrees of one repo silently clobbered each other's state.** The resolver treated every worktree as a shipyard-spawned builder worktree and collapsed them into the parent-repo hash, on the correct premise that `/ship-execute` subagents in `<parent>/.claude/worktrees/<feature>/` must share state with the orchestrator across wave boundaries. Problem: that same branch also fired for hand-made worktrees like `/work/afm-app/trunk3.worktrees/dev` — two humans running parallel Claude sessions on `dev` and `dev2` would read and write the same `sprint.md`, `backlog/`, `config.md`. Locking prevents torn writes but not logical overwrite, so whichever session ran `/ship-sprint` last simply replaced the other's sprint. `bin/shipyard-resolver.mjs` `getProjectRoot()` now classifies worktrees by path shape: a worktree whose realpath'd toplevel is contained in `<parentRepo>/.claude/worktrees/` (with trailing separator, so `.claude/worktrees-other/*` can't masquerade) is a builder worktree and still returns the parent repo root; everything else is a user worktree and returns its own `git rev-parse --show-toplevel`. Three distinct parallel sessions → three distinct hashes → three distinct data dirs.

### Migration
- **Existing user-worktree data is recoverable via the existing orphan-recovery path.** `shipyard-data find-orphans` enumerates worktree paths via `git worktree list` and matches them against `.project-root` breadcrumbs in sibling `projects/<hash>/` dirs, so a user whose data previously lived under the parent-repo-hash will see it surface as a migration candidate in `/ship-init`'s `AskUserQuestion` prompt when they next initialize the worktree. No new code needed — the symmetric case (R1/F5 changed semantics in the opposite direction) already built this recovery path. **Caveat for release notes:** if two user-worktree sessions previously co-mingled state under the parent-repo-hash, only ONE worktree can claim that data via the migration prompt; the other must start fresh. There's no automatic way to de-interleave a shared dir. The `findOrphans` doc comment in `shipyard-data.mjs` now spells this out.

### Tests
- **`tests/test_shipyard_resolver.py`** — existing worktree tests (`test_worktree_returns_parent_repo_root`, `test_worktree_with_claude_project_dir_set`, and their production-scenario mirrors) renamed to `test_builder_worktree_*` and updated to place the worktree under `.claude/worktrees/feat-x`, pinning the builder-shares-parent case explicitly instead of by accident. Added `test_user_worktree_isolates_from_parent` exercising the two-parallel-humans scenario (`trunk3.worktrees/dev` + `trunk3.worktrees/amdb` — three distinct hashes asserted) and `test_user_worktree_with_claude_project_dir_set` + `test_user_worktree_with_claude_project_dir` (production class) for the `CLAUDE_PROJECT_DIR=user-worktree` path. The production-scenarios class now covers both builder and user worktrees with `CLAUDE_PROJECT_DIR` set, mirroring the R14 principle that every resolver behavior must be exercised through the production code path where Claude Code sets the env var.

## [1.0.3] - 2026-04-06

**Fix: `/ship-discuss` session guard bypass + hook diagnostics overhaul.**

### Fixed
- **`/ship-discuss` could implement source code after user approved a plan.** Phase 5 wording let Claude misread the plan-mode payload as an implementation plan instead of a spec-approval summary; user approved, Claude then edited source files during a discussion session. Surfaced by a customer report. Phase 5 is now titled "Spec Approval Gate (NOT an Implementation Plan)" and leads with a tense-based STOP rule ("past-tense outcomes only — no future-tense implementation verbs"). The post-approval cleanup has been split into a new **Phase 6: Finalize** sub-phase, marked mandatory on Approve, with `.active-session.json` deletion pinned as the very last action so the session guard stays live through the entire finalize sequence.
- **Session guard was looking at the wrong path.** `session-guard.py` defaulted to the literal string `.shipyard` when `SHIPYARD_DATA` env was unset, so `.active-session.json` lookup pointed at a phantom directory and the guard never fired. Root cause: `hook-runner.mjs` was silently swallowing `getDataDir({silent: true})` throws in its `catch {}`, leaving the child env with no data dir. Five hook scripts had the same `.shipyard` literal latent bug (`session-guard`, `tdd-check`, `loop-detect`, `on-commit`, `post-compact`).
- **`session-guard.py` absolute-path containment was dead code.** An absolute `SHIPYARD_DATA` was spliced into a relative-path prefix list and could never match. Masked today because `.md` / `.json` extension checks already cover every Shipyard data file, but would have fired on the first non-`.md` write to the data dir during a discuss session.
- **`session-guard` matcher was missing `NotebookEdit`.** hooks.json registered `Write|Edit|MultiEdit`; `auto-approve-data` used the full `Edit|Write|NotebookEdit|MultiEdit` set. Same tool-allowlist drift class CLAUDE.md already warns about — now mirrored.

### Added
- **New `project-files/scripts/_hook_lib.py`** — shared helpers used by all Shipyard hook scripts:
  - `resolve_shipyard_data()` — single fallback pattern for when `SHIPYARD_DATA` env var is unset (env var → `bin/shipyard-resolver.mjs` subprocess → empty string, never raises).
  - `log_breadcrumb(data_dir, log_name, decision, fields)` — uniform append-with-locking + rotation for hook log files. Platform-aware (`fcntl` / `msvcrt`), 256 KB rotation cap, 1000-line tail retained, all errors swallowed so diagnostics never break the hook itself.
  - `sanitize_for_log()` — strips control chars (newlines, ANSI, DEL, all C0/C1), caps length at 256 chars. Applied to every field before writing to the log, defending against log-line forgery via crafted frontmatter and indirect prompt injection through file paths.
  - `data_dir_contains()` — `commonpath`-based containment check, handles Windows drive-cross via `ValueError` → `False`.
- **`.session-guard.log` breadcrumb** — every `session-guard.py` invocation now writes one line with a decision code: `block`, `allow-ext`, `allow-prefix`, `allow-in-data`, `allow-no-session`, `allow-impl-skill`, or `skip`. Absence of the log in `shipyard-context diagnose` output now reliably means "the hook chain never ran", not "the hook ran and allowed".
- **`hook-runner.mjs` stderr warning on resolver throw.** The previous `catch {}` silently swallowed `ShipyardResolverError` and git/fs failures. Now writes a one-line warning with the error class and first 240 chars of the message so the root cause is visible in hook output next recurrence. Pairs with the session-guard breadcrumb for a full diagnostic chain.
- **Compaction Recovery knows about Phase 6.** If auto-compaction interrupts between Approve and Finalize step 5, `ship-discuss` now resumes at the correct Phase 6 step based on BACKLOG.md state instead of treating the partially-finalized feature as mid-Phase-3.
- **REFINE mode has an approval gate.** Added `Step 5: Approval Gate & Finalize` that runs Phase 5 + Phase 6 against the refined feature, with REFINE-specific no-ops documented (status preserved for already-approved features, no BACKLOG duplicate append, idea archival conditional on idea graduation). Pre-existing gap — REFINE previously ended without ever cleaning up `.active-session.json`.
- **`shipyard-data archive-sprint <sprint-id>`** — atomic single-command sprint archival. Renames `$(shipyard-data)/sprints/current/` → `$(shipyard-data)/sprints/<sprint-id>/` in a single `rename(2)` syscall and recreates an empty `current/`. Refuses to overwrite an existing archive dir unless `--force` (with the same safety contract as `migrate`). Sprint ID is validated against a strict `^sprint-[0-9]{3,}$` allowlist to reject path traversal via argv. Ship-sprint (cancel + completed-sprint-resume branches) and ship-review (release-step-3) now route through this command instead of synthesizing raw `cp`/`mv`/`mkdir` bash, which used to trigger permission prompts because the plugin data dir lives outside the project root (Claude Code issue #41763). `ship-sprint` also gained `Bash(shipyard-data:*)` in its frontmatter `allowed-tools`. New eval assertions `uses_atomic_archive_sprint_command` (ship-sprint + ship-review) and `allows_bash_shipyard_data` (ship-sprint) prevent regression.
- **Test-runner subagent captures via `shipyard-logcap run`** instead of `$(shipyard-data)/.test-output*.tmp`. `ship-execute/references/test-delegation.md` and `agents/shipyard-test-runner.md` were instructing the test-runner subagent to redirect test command stdout to files inside the plugin data dir — every `> $(shipyard-data)/…` redirection hit Claude Code's "suspicious path" prompt on every wave. Rewritten to use `shipyard-logcap run <tier> -- <command>` for the single-tier prompt and three separate `run` invocations (one per tier) for the multi-tier sprint-completion variant. `shipyard-logcap` is Shipyard's cross-platform capture primitive (Node implementation with a `.cmd` shim on Windows) that tees output to `$TMPDIR/shipyard/<project-hash>/<session>/` — always in the default Bash allow scope on every platform. Re-analysis via `shipyard-logcap tail` / `grep` replaces manual Read-tool calls on temp files, and built-in rotation eliminates manual cleanup. This is cross-platform correct whereas an earlier draft that used POSIX `mktemp` would have failed on plain Windows cmd.exe / PowerShell (no `mktemp` binary). New assertions `test_delegation_uses_shipyard_logcap`, `test_delegation_no_raw_data_dir_redirect`, and `test_delegation_no_mktemp` enforce it.
- **Removed dead `cp -r` template copy from `/ship-init`.** Lines 246–249 of ship-init/SKILL.md contained a redundant `cp -r "$PLUGIN_ROOT/project-files/templates/"*.md "$(shipyard-data)/templates/"` block that was already covered by `shipyard-data init` (which copies templates via `cpSync` at `shipyard-data.mjs:81–92`). The redundant bash line was triggering a "suspicious path" prompt on every first-time `/ship-init` run despite contributing nothing. Replaced with a prose note explaining that `shipyard-data init` handles templates and explicitly forbidding future raw template-copy bash lines. New assertions `no_redundant_templates_cp` and `relies_on_init_for_templates`.
- **New eval check `check_bash_fenced_portability`** in `tests/eval-run.py` scans every bash code fence inside skill SKILL.md / reference docs and agent bodies for POSIX-only tokens (`mktemp`, `readlink -f`, bare GNU `realpath`, BSD `sed -i ''`, GNU `stat -c`, `/dev/stdin`) and fails CI with a precise `file:line` pointer and fix hint. Complements the existing `check_bash_preexec_portability` which only scans pre-exec `!`cmd`` lines — the new check covers fenced examples that `check_bash_preexec_portability` couldn't see. The gap was exposed by an earlier pass in this same release that used `mktemp -t` in `test-delegation.md` and would have shipped undetected since eval-run doesn't execute on Windows in CI today. Bare `realpath` uses a `(?<![-.])` lookbehind to exclude compound identifiers like `shipyard-realpath`, `fs.realpathSync`, and `my-realpath`. Added a corresponding rule to workspace `CLAUDE.md` under "Cross-Platform: Windows Is a First-Class Target" naming each banned token and pointing at `shipyard-logcap` as the canonical cross-platform capture primitive.

### Changed
- **`session-guard.py` logic ordering.** Extension allow and `data_dir_contains()` now run BEFORE `rel_path` computation so writes inside `SHIPYARD_DATA` (when the data dir lives outside the repo, which is the default plugin-data layout) are correctly classified as `allow-in-data` instead of being dropped as "out of project, skip".
- **`auto-approve-data.py` refactored** to use `_hook_lib` helpers. Stale comment about "PreToolUse fires for ALL Edit/Write tool calls including ones that don't go through hook-runner" corrected — current `hooks.json` routes every write-tool entry through `hook-runner.mjs`, so the resolver fallback is insurance against `getDataDir` throws rather than a primary path. Behavior unchanged.
- **`hook-runner.py` prepends the scripts dir to `sys.path`** before loading the target hook script so siblings can `from _hook_lib import ...`. Previously each script would have had to manipulate `sys.path` on its own.
- **`loop-detect.py`, `on-commit.py`, `tdd-check.py`, `post-compact.py`** now use `_hook_lib.resolve_shipyard_data()` instead of the `.shipyard` string literal. Historical `.shipyard` fallback retained for test environments where the Node resolver isn't reachable.

### Tests
- **New `tests/test_session_guard.py`** — 14 tests covering every decision code, NotebookEdit matcher parity, absolute-path containment for out-of-repo data dirs, sanitization of crafted `.active-session.json` skill fields with embedded newlines, concurrent breadcrumb writes (20 threads, no line loss), resolver fallback when env var is unset, and clean handling of invalid JSON input.
- **New `tests/test_hook_lib.py`** — 17 unit tests for the shared helpers covering `sanitize_for_log`, `data_dir_contains`, `log_breadcrumb` (including rotation), and `resolve_shipyard_data`.
- **Added 9 assertions to `tests/assertions/ship-discuss.json`** pinning the new Phase 5/6 semantics: spec-approval title, STOP rule presence, tense heuristic, failure-mode naming, Phase 6 existence, mandatory-on-approve flag, idea archival placement, guard-delete-is-last ordering, and the updated phase sequence string.
- **Updated `tests/assertions/ship-execute.json`** `workaround_auto_approve_scoped_to_data_dir` now pins the `data_dir_contains(file_path, shipyard_data)` call shape instead of the raw `commonpath` expression (moved into `_hook_lib`).

## [1.0.2] - 2026-04-06

**Fix: `/ship-discuss` permission error + full Windows portability of skill pre-exec blocks.**

### Fixed
- **`/ship-discuss` (and 5 other skills) failed with `Shell command permission check failed for pattern "!shipyard-context path"`** because their `allowed-tools` frontmatter listed no `Bash` scope, yet their SKILL.md bodies used the `!`command`` pre-exec syntax that runs through the Bash tool at load time. Affected: `ship-discuss`, `ship-spec`, `ship-backlog`, `ship-bug`, `ship-sprint`, `ship-help`. Each now declares narrowly-scoped `Bash(shipyard-context:*)` (plus `Bash(shipyard-data:*)` where needed) — the smallest permission that unblocks the pre-exec without granting full shell. Surfaced by a customer report.

### Changed — Windows portability (the real fix)
- **All skill pre-exec blocks are now single-token argv.** Every `!`command`` line in every SKILL.md calls only `shipyard-context`, `shipyard-data`, or `shipyard-logcap`, with alphanumeric/hyphenated arguments. No quoted-with-spaces strings, no shell operators (`|`, `>`, `&&`, `||`, `;`), no external POSIX tools (`head`, `cat`, `python3`, `[`). Root cause: cmd.exe's `%*` mangles quoted arguments containing spaces when they cross a `.cmd` wrapper, silently corrupting context injection on Windows. Keeping every argv token simple eliminates the whole class of bug.
- **`bin/shipyard-context.mjs` gained named subcommands** that keep fallback strings inside the Node CLI rather than passing them through argv:
  - `view <name> [lines]` — reads a registered data file with baked defaults. Registry: `config`, `codebase`, `backlog`, `sprint`, `sprint-progress`, `sprint-handoff`, `metrics`, `data-version`.
  - `list <name> [limit]` — dir/glob/sorted-glob with baked defaults. Registry: `epics`, `features`, `debug-sessions`, `quick-tasks`.
  - `count-of <name>` — count for registered location (`bugs`, `features`, `epics`).
  - `reference <skill> <name> [lines]` — reads `<CLAUDE_PLUGIN_ROOT>/skills/<skill>/references/<name>.md`. Strict allowlist regexes (`^ship-[a-z0-9-]+$` for skill, `^[a-z0-9][a-z0-9._-]*$` for name) plus realpath-containment within the plugin tree — a malicious skill body cannot escape.
  - `version` — reads `plugin.json` via Node. Replaces `cat ${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json | python3 -c "..."` in `/ship-help` which silently produced "version unknown" on Windows (no `python3` in cmd.exe, and `python` naming is inconsistent across Python installers).
  - `project-claude-md [lines]` — reads `<projectRoot>/CLAUDE.md` via the resolver. Replaces `head -50 CLAUDE.md || echo "No CLAUDE.md"` in `/ship-init`.
  - `legacy-check` — checks `<projectRoot>/.shipyard/config.md` via the resolver. Replaces `[ -f .shipyard/config.md ] && echo "LEGACY_SHIPYARD_DETECTED" || echo "NO_LEGACY"` in `/ship-init`. Emits the exact same sentinel tokens so ship-init's branching is unchanged.

  Adding a new pre-exec source is now a one-line registry edit in `shipyard-context.mjs` instead of copying a fallback string into every skill.

- **Skill pre-exec blocks migrated** across all 12 skills:
  - `shipyard-context head config.md 50 NO_CONFIG` → `shipyard-context view config`
  - `shipyard-context head codebase-context.md 50 "No codebase context"` → `shipyard-context view codebase`
  - `shipyard-context ls spec/epics 20 "No epics yet"` → `shipyard-context list epics`
  - `shipyard-context ls-glob "debug/*.md" 5 "No active debug sessions"` → `shipyard-context list debug-sessions`
  - `shipyard-context ls-sort "spec/tasks/Q-*.md" "No quick tasks yet"` → `shipyard-context list quick-tasks`
  - `shipyard-context count spec/bugs` → `shipyard-context count-of bugs`
  - `!head -80 ${CLAUDE_PLUGIN_ROOT}/skills/ship-discuss/references/challenge-surface.md` (three occurrences in `/ship-discuss`) → `!shipyard-context reference ship-discuss challenge-surface 80`
  - `!cat ${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json | python3 -c "..."` → `!shipyard-context version`
  - `!head -50 CLAUDE.md || echo "No CLAUDE.md"` → `!shipyard-context project-claude-md`
  - `![ -f .shipyard/config.md ] && echo LEGACY... || echo NO_LEGACY` → `!shipyard-context legacy-check`
  - `!shipyard-data find-orphans 2>/dev/null || true` → `!shipyard-data find-orphans` (underlying subcommand already exits cleanly; the bashisms were belt-and-braces).

- **New eval check `check_bash_preexec_portability`** in `tests/eval-run.py` hard-fails CI if a future SKILL.md reintroduces any of the three Windows-hostile patterns: quoted argument containing a space, non-portable root command (anything other than the three `shipyard-*` CLIs), or shell operators. Failure messages name the exact fix ("add a subcommand to `shipyard-context`…"). Prevents the regression class from returning silently.

- **New eval check `check_bash_preexec_permissions`** scans every SKILL.md body for `!`cmd`` pre-exec blocks and verifies the root command is covered by a matching `Bash` scope (plain or scoped form) in `allowed-tools`. This is the rule whose absence let the original `/ship-discuss` bug ship — now enforced at eval time.

### Added
- **`tests/test_shipyard_context_cli.py`** — 33 tests covering every new subcommand. Includes security-critical paths: `reference` rejects invalid skill slugs, invalid reference names, slashes in reference names, and path-traversal attempts; fails gracefully with an actionable message when `CLAUDE_PLUGIN_ROOT` is unset. `version` handles missing plugin root and malformed `plugin.json` without throwing. `view` has a registry-walking sanity test that catches a future typo pointing at an unreachable path. One meta-test scans every SKILL.md for quoted-space args so both pytest and eval-run surface the same regression class.

## [1.0.1] - 2026-04-06

**First public release on GitHub.** Earlier `0.x` versions described in prior internal changelogs were never tagged or published; `1.0.1` is the baseline. The sections below document what's in the release. For a feature overview of what Shipyard does, see the README.

### Added
- **Live verification capture via `shipyard-logcap`.** New primitive in `plugins/shipyard/bin/` that wraps an arbitrary verification command, tees raw stdout+stderr to a rotating file in `$TMPDIR/shipyard/<project-hash>/<session>/`, and propagates the child's exit code. Subcommands: `run`, `tail`, `grep`, `list`, `path`, `probe`, `prune`. Cross-platform (Node `.mjs` + symlink-aware `.sh` wrapper + Windows `.cmd` wrapper). **Why it exists:** when reviewing a sprint or debugging a flaky repro, the moment live output passes through a grep or filtered tail the unfiltered stream is gone — if the filter missed a signal, the only recourse is to re-run the command, burning tokens, wall-clock time, and sometimes device/cloud minutes. `shipyard-logcap` lets the orchestrator re-analyze from the captured file instead. The primitive is intentionally dumb (no log parsing, no keyword classification, no auto-tuning); the smart layer is `skills/ship-execute/references/live-capture.md`, which teaches Claude the principle, the re-analysis loop, and a decision table for picking bounds per command profile. `ship-execute` sets `SHIPYARD_LOGCAP_SESSION=<sprint>-wave-<N>` so wave captures group together. Minimum `--max-size` is 64K (one Node pipe high-water mark) so single chunks can't silently overflow the ceiling. Strict allowlist on capture names and session env. `shipyard-context diagnose` now surfaces recent capture activity for bug reports.
- **`/ship-status diagnose` subcommand** — prints the resolver's view (`SHIPYARD_DATA`, project root, project hash, all `CLAUDE_*` env vars, last 20 lines of the auto-approve breadcrumb log). The format customers attach to bug reports about permission prompts or state divergence. Implemented in `shipyard-context.mjs`.
- **Orphan plugin-data recovery in `/ship-init`.** `shipyard-data find-orphans` scans `<plugin-data>/projects/*/.project-root` breadcrumbs and matches them against the current parent repo and its worktrees. When the worktree-detection fix in the resolver changed the project hash for users running Claude Code from inside a standalone worktree, their previous data dir became orphaned at the old hash — one customer had 1.1 MB of real Shipyard state (config, 10+ feature specs, sprints, codebase context) under the old hash that the new resolver couldn't see. `/ship-init` now invokes `find-orphans` alongside the existing legacy-shipyard probe; on hits with no current `config.md`, it prompts via `AskUserQuestion` and runs `shipyard-data migrate <orphan-dir>`. Post-migration, the source path is announced back so the user can clean up the leftover MBs.
- **6 specialized code review agents + investigator**: `shipyard-review-security`, `-bugs`, `-silent-failures`, `-patterns`, `-tests`, `-spec`, plus `shipyard-investigator`. Wave 1 spawns the 6 single-responsibility scanners in parallel (sonnet, fresh 200K each, Read-only); Wave 2 spawns the investigator (opus, fresh per finding) for high-stakes findings. Replaces the monolithic `shipyard-reviewer` which ran 3 sequential review passes in one 200K context and was running out of room on real customer sprints.
- **`shipyard-data with-lock <key> -- <command>` primitive** for serializing writes to shared Shipyard data files: atomic `O_EXCL` lock-file create on POSIX and Windows, `Atomics.wait` for true synchronous sleep (no spin loop), pid liveness via `process.kill(pid, 0)` before stealing stale locks. Documented in `/ship-execute`'s wave-boundary section as the forward-compatible idiom for any future flow that introduces parallel writers.
- **`shipyard-data find-orphans`** subcommand (consumed by orphan recovery above).
- **`shipyard-data migrate <src> [--force]`** atomic single-command migration: tree creation, copy, transient-state cleanup, automatic `.pre-migrate-backup-<ts>` snapshot when `--force` is used, `.project-root` rewrite. Replaces the four-line POSIX shell sequence (`cp -r` / `rm -rf`) that wouldn't work on Windows without Git Bash.
- **Auto-approve permission hook breadcrumb log** at `$SHIPYARD_DATA/.auto-approve.log` — one line per invocation (timestamp, decision, tool, file, data dir). Capped at 1000 lines, rotated at 256KB. Surfaced via `shipyard-context diagnose` so customers debugging permission prompts have the trail.
- New test modules: `test_shipyard_resolver.py` (14 tests), `test_shipyard_data_cli.py` (10 tests), `test_shipyard_logcap.py` (29 tests). Plus `test_auto_approve_data.py` expanded from 12 to 16 tests.

### Changed
- **`shipyard-data` and `shipyard-context` ported from Python to Node** (`shipyard-data.mjs`, `shipyard-context.mjs`) with cross-platform wrappers (`.sh` for POSIX, `.cmd` for Windows). The legacy extensionless Python scripts couldn't run on Windows (no shebang support, not in `PATHEXT`). Skills continue invoking the CLIs as bare commands; `PATH` lookup picks the right wrapper per OS. The `.sh` wrappers are symlink-aware (walk the `readlink` chain before computing the bin dir) so PATH symlinks like `/usr/local/bin/shipyard-data → plugin path` work correctly.
- **`shipyard-resolver.mjs` is now the SINGLE SOURCE OF TRUTH** for project root, project hash, and data dir resolution. The previous codebase had three subtly-different implementations across `shipyard-data`, `hook-runner.py`, and `shipyard-context` that drifted and made the auto-approve hook compute a different `SHIPYARD_DATA` than the skill that triggered the write — silently breaking the permission workaround. The resolver detects worktrees via `git-common-dir` and returns the parent repo root, so all worktrees of one project share state. `CLAUDE_PROJECT_DIR` is used as a starting cwd for git, NOT short-circuited as the answer (production sets it to the worktree path for builder subagents — returning it directly would bypass the worktree fix). Hash format pinned to legacy `sha256(path + '\n')[:12]` so existing customers' non-worktree data dirs remain valid.
- **Data dir discovery is now fail-loud.** When no `CLAUDE_PLUGIN_DATA`, no probe-relative path, and no populated legacy dir can be found, the resolver exits non-zero with an actionable error message naming the env var to set. The previous "helpful" fallback could create a phantom data dir at the wrong location.
- **`ship-review` now uses `model: opus`** and orchestrates the code review pipeline directly. Code review moved out of `ship-execute` into `ship-review`'s new Stage 0 (single responsibility); `ship-execute` now stops at sprint completion. Shared orchestration logic lives in `ship-review/references/code-review-orchestration.md`.
- **README documents worktree data sharing semantics and Windows portability** — explicit note that the project hash is derived from the parent repo root so all worktrees share one data dir, and that `.cmd` wrappers inherit cmd.exe's argument-quoting limits (use `CLAUDE_PLUGIN_DATA` env var for paths with spaces).
- **`ship-execute`** SKILL.md gained a "Live Verification Capture" top-level section and a per-wave `SHIPYARD_LOGCAP_SESSION` export note in Step 2.
- **`ship-review`** SKILL.md Stage 2 now wraps dev-server and E2E observation commands through `shipyard-logcap` and prefers grepping the capture before re-running anything.
- **`ship-bug`** SKILL.md HOTFIX mode now wraps the repro command through `shipyard-logcap`.
- **Hook chain computes `SHIPYARD_DATA` in-process.** `hook-runner.mjs` imports `getDataDir` from the resolver and passes the resolved path via env to the Python child. The Python side reads the env var on the fast path and only subprocess-falls-back in tests. Saves ~80ms per `Edit`/`Write` tool call, ~4 seconds on a 50-edit sprint. Subprocess timeout dropped from 10s to 2s — long hangs are real bugs to fail fast on, not paper over with editor stalls.
- **`/ship-execute` documents the `with-lock` primitive** as the forward-compatible idiom for any future flow that introduces parallel writers (recovery, parallel review fixers, team-mode changes). The race that motivated the primitive is theoretical in current code paths (`PROGRESS.md` is written only by the orchestrator today), but documenting the pattern keeps the building block discoverable.

### Fixed
- **Auto-approve permission hook hardening** (`auto-approve-data.py`, `hooks.json`):
  - Add `MultiEdit` to the auto-approve and session-guard `PreToolUse` matchers (was `Edit|Write|NotebookEdit` only — multi-file edits weren't covered, so the suppression failed for them).
  - Fix `MultilineEdit` → `MultiEdit` typo in the `PostToolUse` loop-detect matcher; the hook had never fired for any multi-edit invocation.
  - Self-resolve `SHIPYARD_DATA` when the env var isn't set (`PreToolUse` fires for non-Shipyard contexts that don't go through `hook-runner`, where the previous code silently no-op'd — defeating the workaround for those calls).
  - Auto-create the data dir before writing the breadcrumb log so the first hook invocation for a brand-new project (pre `/ship-init`) doesn't silently skip logging.
  - Wrap append + rotation in a platform-aware `fcntl`/`msvcrt` lock — POSIX append-mode writes ≤`PIPE_BUF` are atomic for concurrent writers but Windows offers no such guarantee, and the rotation block (read → truncate → rewrite) was racy on every platform.
- **`hook-runner.py` hard-fails when plugin root cannot be located** instead of silently using a guessed path. Silent guessing was how the permission workaround got bypassed in the first place.

### Security
- **Hooks hardened against symlink escape, prompt injection, and path traversal** (`auto-approve-data.py`, `worktree-branch.py`, `loop-detect.py`, `on-commit.py`, `post-compact.py`, `shipyard-context`, `hook-runner.py`):
  - `auto-approve-data.py` uses `realpath` + `commonpath` instead of `abspath` + `startswith`, defending against TOCTOU symlink-based path-escape attacks. Rejects `..` segments before resolution. Dropped `Read` from approved tools (scope reduction).
  - `worktree-branch.py` validates worktree name against strict allowlist (`^[A-Za-z0-9][A-Za-z0-9._-]{0,63}$`, no `.lock` suffix). Containment check on resolved worktree path.
  - `loop-detect.py` and `on-commit.py` sanitize file paths before printing to hook stdout (strip control chars, cap length) to prevent indirect prompt injection via malicious filenames.
  - `post-compact.py` allowlist-validates frontmatter values (sprint id, branch, wave) before echoing to Claude after compaction.
  - `shipyard-context` adds a `safe_join()` helper and validates all relative path arguments against traversal and absolute paths.
  - 12 new unit tests including symlink escape, sibling-dir prefix attack, pre-resolution traversal rejection, and Read scope verification.

### Removed
- **Legacy `shipyard-reviewer` agent** (replaced by 6 specialized `shipyard-review-*` agents + `shipyard-investigator`).
- **Legacy extensionless Python scripts** for `shipyard-data` and `shipyard-context` (replaced by Node `.mjs` + per-OS `.sh`/`.cmd` wrappers).
